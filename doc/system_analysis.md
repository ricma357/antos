# Antos Event-Driven Backtesting Platform: System Analysis & Architecture Guide

This document provides a comprehensive engineering analysis of the **Antos** event-driven backtesting platform. It covers the current architecture, identifies critical logical and financial simulation gaps, and outlines an actionable technical roadmap to resolve them.

---

## 1. System Architecture

The platform is designed around a decoupled, event-driven architecture that models the lifecycle of financial market data, trade generation, and execution.

### Module Overview

- **Event Definitions ([src/events.py](file:///Users/flipis/dev/antos/src/events.py))**:
  Contains the core data containers that pass through the system:
  - [MarketEvent](file:///Users/flipis/dev/antos/src/events.py#L12): Captures OHLCV market updates.
  - [SignalEvent](file:///Users/flipis/dev/antos/src/events.py#L29): Generated by strategies to indicate target positions (`LONG`, `SHORT`, `EXIT`).
  - [OrderEvent](file:///Users/flipis/dev/antos/src/events.py#L43): Sized trade instructions generated by the portfolio manager.
  - [FillEvent](file:///Users/flipis/dev/antos/src/events.py#L58): Broker execution report containing executed prices, fees, and slippage.

- **Data Streaming ([src/data_provider.py](file:///Users/flipis/dev/antos/src/data_provider.py))**:
  - The [HistoricalCSVDataProvider](file:///Users/flipis/dev/antos/src/data_provider.py#L7) parses local daily CSV files, normalizes fields (e.g. using `Adj Close` when available), and merges multi-symbol data points chronologically.
  - To prevent non-deterministic order processing when multiple assets share the same daily timestamp, it applies a secondary sorting key based on the asset ticker symbol.

- **Simulation Loop ([src/engine.py](file:///Users/flipis/dev/antos/src/engine.py))**:
  - The [BacktestEngine](file:///Users/flipis/dev/antos/src/engine.py#L25) manages the main simulation loop.
  - It enforces a strict temporal sequence to eliminate lookahead bias:
    1. **Process pending fills** using the current bar's **Open** price (not yesterday's Close price).
    2. **Mark-to-market** portfolio valuations at the current bar's **Close** price.
    3. **Evaluate strategy rules** to produce signal events.
    4. **Generate and queue sized orders** to be processed at the beginning of the next bar.

- **Portfolio & Risk Management ([src/portfolio.py](file:///Users/flipis/dev/antos/src/portfolio.py))**:
  - The [Portfolio](file:///Users/flipis/dev/antos/src/portfolio.py#L6) handles position bookkeeping, cash reconciliation, and sizing logic.
  - Sizing is calculated dynamically as a fraction of total Net Asset Value (NAV), ensuring scale-invariance as capital changes.
  - Pre-trade risk checks clamp purchase quantities using the function [Portfolio._clamp_buy_quantity](file:///Users/flipis/dev/antos/src/portfolio.py#L56) to prevent cash balance overdrafts.

- **Execution Handler ([src/execution/sim_broker.py](file:///Users/flipis/dev/antos/src/execution/sim_broker.py))**:
  - The [SimulatedBroker](file:///Users/flipis/dev/antos/src/execution/sim_broker.py#L5) acts as an exchange proxy.
  - It holds orders in a queue, executes them at the next bar's Open price, and applies transactional friction (slippage and commission rates).

## 2. Identified Gaps & Simulation Flaws

During code review and execution analysis, several critical bugs, architectural gaps, and math approximations were identified.

### Gap A: Strategy-Portfolio State Desynchronization (Critical Bug)

#### Description
Strategies like [SMACrossover](file:///Users/flipis/dev/antos/src/strategy/sma_crossover.py#L5) and [RSIMeanReversion](file:///Users/flipis/dev/antos/src/strategy/rsi_mean_reversion.py#L7) track their trading states using an internal variable `self._position_state`. This state is updated immediately when a signal is emitted.

However, the [Portfolio](file:///Users/flipis/dev/antos/src/portfolio.py#L6) applies a pre-trade risk constraint: it checks available cash and clamps the order quantity using [Portfolio._clamp_buy_quantity](file:///Users/flipis/dev/antos/src/portfolio.py#L56). If cash is insufficient, the order size is clamped to `0`. 

#### Impact
When cash is clamped to `0`, the strategy believes it is holding a position (`self._position_state = 1`), whereas the [Portfolio](file:///Users/flipis/dev/antos/src/portfolio.py#L6) is flat (`self.positions[symbol] = 0`). This causes:
1. The strategy to ignore all subsequent entry signals (since it believes it is already in the market).
2. The strategy to wait for an exit signal, which will do nothing when processed by the portfolio (due to `current_qty == 0`).
3. Total divergence between backtest assumptions and actual portfolio execution.

---

### Gap B: Cash Over-Allocation in Multi-Asset Environments (Critical Bug)

#### Description
In [BacktestEngine.run](file:///Users/flipis/dev/antos/src/engine.py#L53), the loop iterates through merged chronological events. When multiple symbols share the same day (e.g. `2020-01-02` for both `SPY` and `BTC-USD`), they are processed sequentially within the same bar iteration.
If both symbols generate a `LONG` signal:
1. `SPY` generates a signal. [Portfolio.generate_order](file:///Users/flipis/dev/antos/src/portfolio.py#L85) checks available cash, verifies it has enough cash, and queues a buy order for `SPY`.
2. `BTC-USD` generates a signal. [Portfolio.generate_order](file:///Users/flipis/dev/antos/src/portfolio.py#L85) checks the **same** available cash (since cash is only deducted when the order is filled at the next bar's Open), verifies it has enough cash, and queues a buy order for `BTC-USD`.
3. On the next bar, both orders are executed. The cash balance is deducted for both fills in [Portfolio.update_fill](file:///Users/flipis/dev/antos/src/portfolio.py#L188). 

#### Impact
Because cash was checked before either order was filled, the total cost of the queued orders can exceed the available cash balance. This causes `self.cash` to go negative, violating the strict cash-account constraint.

### Gap C: Dummy Limit Order Execution

#### Description
The [OrderEvent](file:///Users/flipis/dev/antos/src/events.py#L43) structure contains fields for `order_type` (`'MKT'` or `'LMT'`) and `price`. However, the [SimulatedBroker](file:///Users/flipis/dev/antos/src/execution/sim_broker.py#L5) implements execution in [SimulatedBroker.process_market_event](file:///Users/flipis/dev/antos/src/execution/sim_broker.py#L34) by blindly filling **all** pending orders at the next bar's Open price:
```python
base_price = event.open_price
# ...
fill_price = base_price + (direction_multiplier * slippage_offset)
```

#### Impact
Any limit order is treated as a market order. If a strategy issues a Limit Buy order below the market price, the broker fills it immediately at the next Open price, regardless of whether the asset price ever touched the target limit price.

---

### Gap D: Fixed Annualization Assumptions in Metrics Calculation

#### Description
In [BacktestEngine._build_result](file:///Users/flipis/dev/antos/src/engine.py#L84), performance metrics are annualized using a hardcoded value of `252` (standard equity trading days per year):
```python
sharpe = (avg_ret / std_ret * (252 ** 0.5)) if std_ret > 0 else 0.0
```

#### Impact
This assumption fails when:
1. Backtesting cryptocurrency assets (e.g. `BTC-USD`), which trade 365 days a year.
2. Backtesting mixed portfolios (cryptocurrency + equities), where the daily frequency is non-uniform.
3. The dataset spans less than a year or contains non-standard trading frequency gaps.
Additionally, the Sharpe ratio calculations assume a Risk-Free Rate of `0.0`, which does not reflect realistic macro environments.

---

### Gap E: Approximate RSI Moving Average Calculation

#### Description
In [RSIMeanReversion._compute_rsi](file:///Users/flipis/dev/antos/src/strategy/rsi_mean_reversion.py#L52), the averages of gains and losses are computed as standard simple moving averages:
```python
avg_gain = sum(gains[-self.period:]) / self.period
avg_loss = sum(losses[-self.period:]) / self.period
```

#### Impact
Wilder's original RSI requires Wilder's Smoothed Moving Average (SMMA / Modified Moving Average), which is mathematically equivalent to an Exponential Moving Average (EMA) with `alpha = 1 / period`. Using standard SMAs causes the RSI values to deviate from industry-standard definitions (such as those in TA-Lib), potentially shifting entry/exit signal timings.

## 3. Recommended Improvement Roadmap

To turn **Antos** into a production-grade, highly reliable research engine, the following modifications are recommended:

### Phase 1: Fix Core Simulation Bugs
1. **Stateless Strategy Signal Evaluation**:
   - Refactor [BaseStrategy.calculate_signals](file:///Users/flipis/dev/antos/src/strategy/base.py#L15) to receive the portfolio's current position sizes, or make the strategy purely stateless (returning target signal indicators) and let [Portfolio](file:///Users/flipis/dev/antos/src/portfolio.py#L6) track position changes.
   - Alternatively, pass the current portfolio positions dictionary `positions` or a query interface to `calculate_signals`.

2. **Pending Cash Tracking**:
   - Introduce a `self.pending_cash_allocated` tracker in [Portfolio](file:///Users/flipis/dev/antos/src/portfolio.py#L6).
   - When an order is generated, reduce the virtual buying power by the estimated transaction value.
   - Reconcile this virtual allocation once the `FillEvent` is processed or when an order is cancelled.

### Phase 2: Enhance Execution Modeling
1. **True Limit Order Matching**:
   - Update [SimulatedBroker.process_market_event](file:///Users/flipis/dev/antos/src/execution/sim_broker.py#L34) to check `order.order_type`.
   - For Limit orders, verify if `order.price` falls within the current bar's range:
     - For Limit Buy: `event.low_price <= order.price`.
     - For Limit Sell: `event.high_price >= order.price`.
   - Fill the order at the limit price (or the Open price if it gapped past the limit price) instead of blindly using the Open price.

2. **Short Borrowing Fees**:
   - Add a cost-of-borrowing rate to the execution and portfolio calculations for negative asset positions.

### Phase 3: Financial Analytics Refinements
1. **Dynamic Annualization**:
   - Analyze the average frequency of dates in the equity dataframe to compute the correct annualization factor (e.g. 252 for stocks, 365 for crypto, 52 for weekly data).
2. **Risk-Free Rate Input**:
   - Allow setting a risk-free rate parameter in [BacktestEngine](file:///Users/flipis/dev/antos/src/engine.py#L25) to calculate Excess Return Sharpe/Sortino ratios.
3. **Standardize RSI Smoothing**:
   - Implement Wilder's SMMA formula in [RSIMeanReversion](file:///Users/flipis/dev/antos/src/strategy/rsi_mean_reversion.py#L7) to align outputs with industry-standard charting libraries.
